The 5 Dependency Rings

Standard Model of Code - Dependency Depth (Ω)

Rings answer the question: "How deep in the dependency graph is this code?" They measure the concentric layers of a clean architecture, from the innermost pure core (R0) to the outermost framework integration (R4).

The fundamental law is: dependencies flow inward. Outer rings can depend on inner rings, but inner rings must never import from outer rings. This ensures the core business logic remains independent of frameworks and I/O.

This model aligns with Clean Architecture, Hexagonal Architecture, and Onion Architecture - all expressing the same principle through different metaphors.

THE DEPENDENCY RULE
Dependencies can only point inward. Ring N may import from Ring N-1, N-2, etc.
Ring 0 never imports from Ring 1 or higher. The core is independent of everything.
R0
Core
No internal deps
The pure center - utilities, constants, and base types with zero internal dependencies. This code is so fundamental it can be copied to any project.
Typical Contents
StringUtils, MathHelpers Constants, Enums Base Error types Pure validation functions
Key Takeaway
R0 code should be pure, stateless, and universally reusable. If it imports anything from your project, it's not R0.
Example Paths
src/core/utils/ src/shared/constants/ lib/primitives/
Common Confusion
R0 vs R1: R0 has no business meaning; R1 contains domain concepts. "User" is R1; "capitalize()" is R0.
R1
Domain
Imports R0 only
The business heart - domain entities, value objects, and business rules. This is what your application is actually ABOUT.
Typical Contents
User, Order, Product entities Email, Money value objects Domain events Business rule validators
Key Takeaway
R1 defines WHAT your system does, not HOW. It knows nothing about databases, HTTP, or frameworks.
Example Paths
src/domain/entities/ src/domain/valueobjects/ src/core/models/
Common Confusion
R1 vs R2: R1 is nouns (User, Order); R2 is verbs (CreateUser, ProcessOrder).
R2
Application
Imports R0-R1
The orchestration layer - use cases, application services, and workflows. This coordinates domain objects to achieve user goals.
Typical Contents
CreateUserUseCase OrderProcessingService WorkflowOrchestrator Command/Query handlers
Key Takeaway
R2 defines the USE CASES - what users can do. It orchestrates R1 entities but uses R3 interfaces (not implementations).
Example Paths
src/application/usecases/ src/application/services/ src/usecases/
Common Confusion
R2 vs R3: R2 defines interfaces (IUserRepository); R3 implements them (PostgresUserRepository).
R3
Adapter
Imports R0-R2
The I/O boundary - repositories, gateways, and external system adapters. This is where the application meets the outside world.
Typical Contents
PostgresUserRepository StripePaymentGateway S3FileStorage EmailSender, SMSClient
Key Takeaway
R3 implements R2's interfaces with real I/O. Swap Postgres for MongoDB by changing R3, not R1 or R2.
Example Paths
src/infrastructure/persistence/ src/adapters/ src/gateways/
Common Confusion
R3 vs R4: R3 adapts external services; R4 handles external requests (HTTP, CLI).
R4
Framework
Imports R0-R3 + externals
The entry point layer - web handlers, CLI commands, and framework integrations. This is where external requests enter your system.
Typical Contents
UserController, OrderRoutes CLI commands (create-user) GraphQL resolvers main.py, index.ts
Key Takeaway
R4 is the thinnest layer - just routing and DI wiring. Business logic belongs in R1-R2, not in controllers.
Example Paths
src/api/controllers/ src/cli/commands/ src/web/routes/
Common Confusion
Fat controllers: If your controller has business logic, move it to R2. Controllers should only translate HTTP → UseCase → HTTP.
