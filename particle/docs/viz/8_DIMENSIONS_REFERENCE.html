The 8 Semantic Dimensions

Standard Model of Code - Orthogonal Classification Axes

Every code entity can be described along 8 orthogonal dimensions. Each dimension answers a different question about the entity, and together they form its complete semantic fingerprint.

The dimensions are orthogonal - knowing the value of one dimension tells you nothing about the others. A function can be any Role, at any Layer, with any Effect. This independence allows rich classification without explosion of categories.

The first dimension (WHAT) uses 200 atom types. The third dimension (Role) uses 41 canonical roles. Other dimensions have smaller, fixed domains.

D1
WHAT
What is this made of?
The structural atom type - what syntactic/semantic category this entity belongs to. This is the primary classification axis.
Domain (200 atoms)
Function Class Interface Module SourceFile +195 more
Key Takeaway
WHAT is determined by syntax and structure. A class is a class regardless of what it does.
Common Confusion
WHAT vs Role: WHAT is the noun (Function); Role is the verb (Validator). Same WHAT can have different Roles.
D2
Layer
Where does this live?
The architectural stratum - which horizontal slice of the architecture this entity inhabits.
Domain (5 values)
Interface Application Core Infrastructure Test
Key Takeaway
Layers are horizontal slices. Each layer has specific responsibilities and allowed dependencies.
Common Confusion
Layer vs Ring: Layers are horizontal (UI/App/Core); Rings are concentric (dependency depth).
D3
Role
What is its purpose?
The functional purpose - what job this entity performs in the system. The "why" of code.
Domain (41 roles)
Query Command Factory Repository Service Handler +35 more
Key Takeaway
Role is about PURPOSE. A function named "validateEmail" has Role=Validator regardless of its implementation.
Common Confusion
Multiple roles: One function can serve multiple roles, but classify by PRIMARY purpose.
D4
Boundary
Does it cross boundaries?
The I/O profile - whether this entity interacts with the outside world and in which direction.
Domain (4 values)
Internal Input Output I/O
Key Takeaway
Boundary code requires extra care - validation, error handling, and security. Internal code can trust its inputs.
Common Confusion
I/O vs Input+Output: I/O means both directions in one operation (e.g., database query).
D5
State
Does it maintain state?
The memory profile - whether this entity holds state between invocations or is stateless.
Domain (2 values)
Stateful Stateless
Key Takeaway
Stateless code is easier to test, scale, and reason about. State should be explicit and minimized.
Common Confusion
Stateful vs uses-state: A function that reads from a database is Stateless; it doesn't OWN the state.
D6
Effect
Does it have side effects?
The purity profile - whether and how this entity modifies state or performs I/O.
Domain (4 values)
Pure Read Write ReadModify
Key Takeaway
Pure functions are deterministic and testable. Push side effects to the edges of your system.
Common Confusion
Read vs Pure: Pure has NO external dependencies; Read accesses external state but doesn't modify it.
D7
Lifecycle
What lifecycle phase?
The temporal profile - when in an object's lifecycle this code runs.
Domain (3 values)
Create Use Destroy
Key Takeaway
Most code is "Use". Create/Destroy code (constructors, cleanup) needs special attention for resource management.
Common Confusion
Create vs Factory: Create is a lifecycle phase; Factory is a Role. Factories produce Create-phase code.
D8
Trust
Classification confidence?
The confidence score - how certain the classifier is about all other dimension values.
Domain (continuous)
0% 25% 50% 75% 100%
Key Takeaway
Trust is meta-information. Low trust means the entity might be misclassified and needs human review.
Common Confusion
Trust vs code quality: High trust means confident classification, not that the code is good.
