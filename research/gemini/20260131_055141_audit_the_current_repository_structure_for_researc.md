### Naming Conventions & Critique of 'archive'

*   **Existing Naming Conventions:**
    *   `src/`: For core Python source code. Standard.
    *   `schema/`: For formal definitions (constraints, profiles). Well-defined and appropriate.
    *   `archive/data/`: Currently used for static, foundational datasets like `1440_csv.csv`. This use of "archive" implies a collection of historical or infrequently modified data.
    *   `.collider/`: For temporary or generated analysis outputs. This is a common pattern for tool-specific output.
*   **Is 'archive' the right name?**
    *   For the `1440_csv.csv` which appears to be a static dataset underpinning the "1440 grid" and thesis, `archive` is an acceptable name, implying a non-active, historical data collection.
    *   However, for *active research materials*, academic papers, literature reviews, or evolving research notes, `archive` can be misleading. It implies something that is no longer in active use or development.
    *   `library` would suggest a collection of reusable resources (code library, component library) or reference materials (literature library). While "library" is better than "archive" for *reference papers*, it can still be ambiguous if it also contains *active* research output (e.g., Jupyter notebooks, draft papers).

### Recommended Canonical Structure

Given the nature of the project (defining a "Standard Model"), a dedicated top-level directory for research and documentation is crucial to clearly separate the theoretical work from the implementation. I recommend `research/` as the primary container for all academic and scientific artifacts, alongside standard `src/` and `schema/` directories.

│   └── ...

**Canonical Locations Summary:**

1.  **Academic references and papers:**
    *   **Recommendation:** `research/papers/`
    *   This directory should hold the primary academic outputs of the project, such as the `THESIS_CONSTRAINT_VALIDATION.md` and `ORPHAN_TAXONOMY.md`. For external papers that are *cited*, `research/references/` is more appropriate.

2.  **Processed document outputs (markdown, chunks):**
    *   **Recommendation:** `research/data/processed/`
    *   This is the ideal place for processed research datasets like `1440_csv.csv` that are directly related to the academic work.
    *   *Note:* The analysis outputs like `output_llm-oriented_*.json` and `output_human-readable_*.html` generated by the `full_analysis.py` tool should remain in the `.collider/` directory or the specified output directory, as they are runtime artifacts of the *tool*, not source research data.

3.  **Research notes and analysis:**
    *   **Recommendation:** `research/notes/`
    *   This provides a dedicated space for informal notes, experimental logs, drafts of papers, Jupyter notebooks, and other intermediary analysis documents.

This structure clearly delineates the "Standard Model of Code" as a research-driven software project, making it easy to navigate its theoretical underpinnings, supporting data, and generated application artifacts.
