8 stages from semantic file to pixels — <25ms total budget

03 — Semantic Files (The Briefs)

Purpose-driven contracts — why each data point exists

A Brief is a YAML file that declares what a dashboard element shows, why it exists, and how it should behave under failure. The engine reads Briefs at startup and uses them to drive every rendering decision.

id: journal.velocity
purpose:
  answers: "How productive was this day?"
  becoming: "Real-time development intensity gauge"
  relevance: 0.95
  attention_cost: glance
  narrative_role: anchor
sense:
  source: local
  endpoint: journal
controls:
  - id: refresh
    action: POST /api/v1/journal/rerun
constraints:
  - "contrast >= 4.5:1"
resilience:
  fallback: last_known_good

The 6 Questions Every Brief Answers

graph LR
    B[Brief] --> Q1["What? (id + sense)"]
    B --> Q2["Why? (purpose.answers)"]
    B --> Q3["How important? (relevance)"]
    B --> Q4["How fast? (attention_cost)"]
    B --> Q5["What if fail? (resilience)"]
    B --> Q6["What rules? (constraints)"]

Key insight: The Brief is a contract, not configuration. It tells the engine the semantic intent so that the Creative department (Art Director, Architect) can make layout decisions automatically based on relevance, attention cost, and narrative role.

04 — The Constraint System (Legal Dept)

Must / Should / Prefer — three tiers of coherence rules

Every token and layout decision passes through the Compliance Officer before reaching the renderer. Constraints are classified into three tiers that determine what happens on violation.

graph LR
    TOKEN[Resolved Token] --> MUST{Must?}
    MUST -->|pass| SHOULD{Should?}
    MUST -->|fail| BLOCK[REFUSE]
    SHOULD -->|pass| PREFER{Prefer?}
    SHOULD -->|fail| FIX[Auto-Correct]
    PREFER -->|pass| RENDER[Render]
    PREFER -->|fail| WARN[Warn + Render]
    FIX --> RENDER

Constraint Types

TierBehavior on FailExample
MUSTRefuse to rendercontrast >= 4.5:1 (WCAG AA)
SHOULDAuto-correct silentlyfont-size >= 12px
PREFERWarn, render anywayborder-radius: consistent

The Auditor logs every constraint evaluation. When a MUST fails, the Auditor produces a human-readable explanation of why the render was blocked and what the developer needs to change. No silent failures.

05 — Anti-Fragility (Risk Dept)

Freeze-on-failure, domain isolation, circuit breakers

The engine treats design domains as isolated failure zones. If the color system breaks, spacing and layout continue unaffected. The Risk Manager enforces domain boundaries and triggers freeze-on-failure when a domain produces invalid output.

graph TD
    subgraph Color Domain
        C1[Color tokens]
        C2[Color constraints]
    end
    subgraph Spacing Domain
        S1[Spacing tokens]
        S2[Spacing constraints]
    end
    subgraph Layout Domain
        L1[Layout rules]
    end
    C1 -.->|"fails: freeze"| C2
    S1 --> S2
    L1 --> RENDER[Publisher]
    S2 --> RENDER
    C2 -.->|"frozen values"| RENDER

Resilience Strategies

Freeze-on-Failure

When a domain produces invalid output, the engine freezes the last known good values for that domain. The rest of the pipeline continues with fresh data. The user sees stale-but-correct rather than broken.

Circuit Breakers

After 3 consecutive failures in a domain, the circuit opens. The domain stops being evaluated entirely until a manual reset or successful health check. This prevents cascading resource waste.

07 — Research Findings

5 deep research queries — 3 unexpected discoveries

Five deep research queries across parametric UI, constraint-driven layout, and agent-to-UI compilation. Three findings stood out as directly applicable.

A2UI by Google

Agent-to-UI framework that compiles agent intent into component trees. Closest prior art to the Dashboard Engine's Brief-to-layout pipeline. Key difference: A2UI is LLM-driven per-request; our engine is deterministic with pre-compiled Briefs.

AGENT-UI GOOGLE RESEARCH

Zag.js

Headless state machines for UI components. Each component is a finite state machine with typed transitions. Validates our approach: the Operations department (Dispatcher, Change Manager) should use FSM patterns internally.

STATE MACHINES APPLICABLE

RL-Based Layout Optimization

Reinforcement learning applied to dashboard layout scoring. Trains on user engagement metrics to optimize widget placement. Maps directly to our Art Director role — but V1 uses heuristic scoring, with RL as a future upgrade path for the Art Director's scoring algorithm.
